## 1. Назначение и актуальность

### Что это такое

Система представляет собой программно-аппаратный комплекс, который в режиме реального времени:

- **считывает** данные с датчиков температуры воздуха (DHT11) и влажности почвы (YL-69);
- **отображает** эти данные в удобном графическом интерфейсе;
- **автоматически управляет** исполнительными механизмами: вентилятором (через реле) и системой полива (через сервопривод);
- **управляет освещением** (RGB-светодиод) по настраиваемому расписанию с возможностью ручного переопределения;
- **сохраняет** все пороговые значения и расписание между перезапусками;
- **упаковывается** в один исполняемый файл `.exe` для Windows без необходимости установки Python.

### Актуальность

Выращивание микрозелени набирает популярность как коммерческий и домашний вид деятельности. При вертикальном выращивании критически важно:

- поддерживать оптимальную температуру (как правило, 18–25 °C) — перегрев тормозит рост и провоцирует болезни;
- не допускать пересыхания почвы — микрозелень чувствительна к дефициту влаги;
- соблюдать световой режим — большинству культур требуется 12–16 часов света в сутки.

Ручной контроль этих параметров трудоёмок и чреват ошибками. Данная система полностью автоматизирует рутинные задачи и освобождает оператора от постоянного присутствия.

## 2. Состав аппаратной части

| Компонент | Модель / тип | Роль |
|---|---|---|
| Микроконтроллер | Arduino Mega 2560 | Центральный узел: чтение датчиков, управление исполнительными механизмами, связь с ПК по USB |
| Датчик температуры/влажности | DHT11 | Измерение температуры воздуха (0–50 °C) и относительной влажности воздуха (20–80%) |
| Датчик влажности почвы | YL-69 + компаратор | Измерение электропроводности почвы, пропорциональной её влажности (0–1023 ADC) |
| Вентилятор охлаждения | 12 В DC + реле | Охлаждение при превышении порога температуры |
| Сервопривод | Micro servo (SG90 или аналог) | Открытие/закрытие клапана системы полива |
| RGB-светодиод | Общий катод или общий анод | Освещение посевов по расписанию |
| ПК / ноутбук | Windows 10/11 | Запуск приложения, отображение GUI, хранение настроек |
| USB-кабель | USB Type-B (Arduino) | Питание Arduino + последовательный порт для обмена данными |

### Почему Arduino Mega 2560, а не Uno?

На Arduino Mega 2560 значительно больше аналоговых входов (16 против 6), цифровых пинов (54 против 14) и аппаратных таймеров (Timer 0, 1, 2, 3, 4, 5). Это позволяет без конфликтов одновременно использовать библиотеку `Servo` (занимает Timer 1) и `analogWrite` на пинах 2, 3, 5 (управляются Timer 3). На Arduino Uno Servo блокирует ШИМ-выходы, что делало бы RGB-управление невозможным.

### Потоки данных

1. **Датчики → Arduino → ПК**: каждые 2 секунды Arduino считывает YL-69 и DHT11, формирует JSON-строку и отправляет по UART. Фоновый поток `serial-read` в `SerialBoard` принимает строки и сохраняет последний `SensorReading`.

2. **ПК → Arduino**: при необходимости включить вентилятор, изменить угол сервопривода или яркость светодиода `Controller` отправляет текстовую команду (`DOUT 7 1`, `SERVO 9 90`, `PWM 2 255`).

3. **GUI ↔ Controller**: `Controller` — `QObject` с Qt-сигналами. При каждом обновлении данных он генерирует сигналы (`sensor_updated`, `fan_state_changed` и т.д.), которые принимают панели GUI и обновляют отображение. Настройки из GUI передаются обратно через `settings_applied` → `SettingsStore.save()` → `Controller.reload_settings()`.

### Многозадачность

Приложение использует **один поток** с двумя кооперирующими циклами событий:
- **asyncio** (через `qasync`) — для цикла опроса датчиков и управления (`Controller.run()`);
- **Qt Event Loop** — для обработки событий GUI.

Фоновый **системный поток** (`serial-read`) занят только блокирующим чтением байт из COM-порта — он не взаимодействует с GUI напрямую, а лишь обновляет `_latest` через мьютекс.

### Почему asyncio, а не threading?

При использовании `threading` каждый поток требует явной синхронизации (Lock, Queue) и не может напрямую взаимодействовать с GUI без `QMetaObject.invokeMethod`. `asyncio` с `qasync` объединяет оба цикла событий в один, что делает код линейным и читаемым. Единственный блокирующий участок — чтение байт из UART — вынесен в отдельный демон-поток и изолирован мьютексом.

### Почему кастомный скетч вместо StandardFirmata?

`StandardFirmata` — универсальный протокол, но он **не поддерживает DHT11**. DHT11 использует собственный однопроводной протокол с точными таймингами. `python-firmata2` не может его обслуживать через Firmata. Кастомный скетч (`grow_control.ino`) читает DHT11 нативно через библиотеку Adafruit, а общение с ПК ведёт через простой текстовый JSON-протокол, легко расширяемый и отлаживаемый.

## 6. Серийный протокол Arduino

Обмен ведётся по USB-UART на скорости **115 200 бод**, строками, разделёнными символом `\n`.

### Последовательность установки соединения

```
[Arduino reset при открытии COM-порта]
      │
      ▼  (2 секунды задержки — стабилизация питания)
Arduino  ──►  "GROW_CTRL_READY\n"
      │
      ▼  (SerialBoard подтверждает успешное подключение)
[Обмен данными]
```

### Данные от Arduino → ПК (каждые 2 секунды)

```json
{"s":450,"t":23.6,"h":49.0}
```

| Поле | Тип | Описание | Диапазон |
|---|---|---|---|
| `s` | integer | Сырое значение АЦП влажности почвы | 0–1023 |
| `t` | float | Температура воздуха, °C | 0–50 (DHT11) |
| `h` | float | Относительная влажность воздуха, % | 20–80 (DHT11) |

> Поля `t` и `h` **отсутствуют**, если DHT11 вернул ошибку чтения (`NaN`). Приложение корректно обрабатывает этот случай.

### Команды ПК → Arduino

| Команда | Пример | Действие |
|---|---|---|
| `DOUT <pin> <0\|1>` | `DOUT 7 1` | Установить цифровой выход (реле вентилятора) |
| `SERVO <pin> <angle>` | `SERVO 9 90` | Повернуть сервопривод на угол 0–180° |
| `PWM <pin> <duty>` | `PWM 2 255` | Установить ШИМ-скважность 0–255 (управление LED) |

## 7. Бизнес-логика управления

### 7.1 Управление вентилятором (гистерезис по температуре)

Используется **гистерезис** для предотвращения частых включений/выключений при колебаниях температуры вблизи порога.

```
Вентилятор ВЫКЛ:                Вентилятор ВКЛ:
                                 
       │← гистерезис→│           │
       ▼             ▼           ▼
[threshold - hyst] [threshold]  [включился]
       
  Если T > threshold            → вентилятор ВКЛ
  Если T < threshold - hyst     → вентилятор ВЫКЛ
  Иначе                         → состояние не меняется
```

**Пример:** порог 28 °C, гистерезис 1 °C:
- при T = 29 °C — вентилятор включается
- при T = 27.4 °C — вентилятор выключается (27.4 < 28 - 1 = 27)
- при T = 27.6 °C — вентилятор остаётся включённым

### 7.2 Управление сервоприводом (гистерезис по влажности почвы)

Аналогичный принцип, но для нижнего порога влажности:

```
  Если soil_pct < threshold            → сервопривод ОТКРЫТ (полив)
  Если soil_pct > threshold + hyst     → сервопривод ЗАКРЫТ (стоп полива)
  Иначе                                → состояние не меняется
```

**Пример:** порог 40%, гистерезис 5%:
- при 35% — сервопривод открывается
- при 46% — сервопривод закрывается (46 > 40 + 5 = 45)

### 7.3 Перевод сырого значения АЦП в проценты влажности почвы

YL-69 выдаёт **меньшее значение при большей влажности** (повышенная проводимость = меньше сопротивление = меньше напряжение на делителе). Формула перевода:

```
soil_pct = 100 × (adc_dry − adc_raw) / (adc_dry − adc_wet)
```

Где:
- `adc_wet` — значение АЦП в воде (калибровочное, типично 200–400)
- `adc_dry` — значение АЦП на воздухе (калибровочное, типично 800–1000)
- `adc_raw` — текущее значение

Результат фиксируется в диапазон [0%, 100%].

### 7.4 Расписание RGB-светодиода

LED управляется по временному расписанию с учётом **часового пояса** (`TIMEZONE` в `.env`). Расписание задаётся парой времён `led_on_time` и `led_off_time`.

`LedScheduler` корректно обрабатывает **ночные окна** (когда время включения больше времени выключения):

```python
# Дневное окно (08:00 → 20:00):
on_time <= current < off_time

# Ночное окно (20:15 → 06:30): переход через полночь
current >= on_time  OR  current < off_time
```

**Ручное переопределение** (Force ON / Force OFF / Auto):
- `None` (Auto) — управление по расписанию
- `True` (Force ON) — всегда включён
- `False` (Force OFF) — всегда выключён

Состояние ручного переопределения сохраняется в `settings.json` и восстанавливается при следующем запуске.

### 7.5 Автоматическое переподключение

Если соединение с Arduino прерывается (отключение USB, перезагрузка платы), `Controller` автоматически пытается переподключиться каждые **5 секунд**. GUI отображает статус соединения в строке состояния.

## 8. Конфигурация (.env)

Создайте файл `.env` в директории рядом с `grow_control.exe` (для продакшена) или в корне проекта (для разработки). Шаблон: `.env.example`.

```ini
# ─── Логирование ─────────────────────────────────────────────────────────────
# Уровень детализации: DEBUG | INFO | WARNING | ERROR | CRITICAL
LOG_LEVEL=INFO

# Дублировать логи в GUI-панель (требует включения кнопкой "Logs" в интерфейсе)
LOG_TO_GUI=false

# ─── Режим работы ────────────────────────────────────────────────────────────
# true  — симуляция (без физического Arduino; для разработки и демонстрации)
# false — реальное железо
MOCK_HARDWARE=false

# ─── Последовательный порт ───────────────────────────────────────────────────
# AUTODETECT — автоматический поиск Arduino по описанию драйвера
# COM3       — явное указание порта
SERIAL_PORT=AUTODETECT
SERIAL_BAUD=115200

# ─── Назначение пинов Arduino ────────────────────────────────────────────────
PIN_DHT=4           # DHT11 DATA → D4
PIN_SOIL_ANALOG=1   # YL-69 AO → A1 (цифра 1 = A1)
PIN_FAN_RELAY=7     # Реле вентилятора → D7
PIN_SERVO=9         # Сервопривод → D9
PIN_LED_R=2         # Красный канал LED → D2
PIN_LED_G=3         # Зелёный канал LED → D3
PIN_LED_B=5         # Синий канал LED → D5

# Тип LED: false = общий катод (common cathode), true = общий анод (common anode)
LED_COMMON_ANODE=false

# ─── Калибровка датчика почвы ────────────────────────────────────────────────
# Измерьте значение АЦП когда датчик в воде (wet) и на воздухе (dry).
# Откройте Serial Monitor в Arduino IDE (115200 baud) и смотрите поле "s".
SOIL_ADC_WET=300    # типично: 200–400
SOIL_ADC_DRY=900    # типично: 800–1000

# ─── Периодичность опроса ────────────────────────────────────────────────────
SENSOR_POLL_MS=1000

# ─── Часовой пояс (IANA) ─────────────────────────────────────────────────────
TIMEZONE=Europe/Minsk
```

> **Важно**: файл `.env` никогда не коммитится в git (добавлен в `.gitignore`). Каждый пользователь создаёт свой файл из шаблона `.env.example`.

## 12. Схемы подключения компонентов

### DHT11 (датчик температуры и влажности)

```
Arduino Mega            DHT11 (вид спереди, 4 пина)
─────────────           ────────────────────────────
5V ─────────────────── Pin 1 (VCC)
                        Pin 2 (DATA) ─── D4
                              │
                        10 kΩ │ (подтягивающий резистор)
                              │
5V ─────────────────────────────── (второй конец резистора)
GND ────────────────── Pin 4 (GND)
                        Pin 3 — не подключён

Примечание: некоторые модули DHT11 имеют 3 пина (VCC, DATA, GND) и
встроенный резистор — тогда 10 кΩ не нужен.
```

### YL-69 (датчик влажности почвы)

```
Arduino Mega            YL-69 (модуль компаратора)
─────────────           ──────────────────────────
5V ─────────────────── VCC
GND ────────────────── GND
A1 ─────────────────── AO  (аналоговый выход — чем суше, тем больше)
(не подключён) ──────── DO  (цифровой выход — не используется)

Зонд (две металлические пластины) подключается к разъёму на модуле.
```

### Реле 5 В + вентилятор 12 В

```
Arduino Mega            Модуль реле           Вентилятор
─────────────           ─────────────         ─────────────────
5V ─────────────────── VCC
GND ────────────────── GND
D7 ─────────────────── IN (сигнальный)

                        COM ─────────────── +12 В источника питания
                        NO ──────────────── Красный провод вентилятора
                        Чёрный провод вентилятора ──── GND источника

HIGH на D7 → реле замыкается → вентилятор вращается
LOW  на D7 → реле размыкается → вентилятор стоит
```

### Micro Servo (SG90 и совместимые)

```
Arduino Mega            Servo SG90
─────────────           ─────────────────────
5V ─────────────────── Красный (питание)
GND ────────────────── Коричневый/чёрный (земля)
D9 ─────────────────── Оранжевый/жёлтый (сигнал PWM)

angle = 0°   → клапан закрыт (нет полива)
angle = 90°  → клапан открыт (полив)
```

### RGB-светодиод (общий катод)

```
Arduino Mega            RGB LED (4 ноги)
─────────────           ────────────────────────────────
D2 ──── 220 Ω ──────── R (красный анод)
D3 ──── 220 Ω ──────── G (зелёный анод)
D5 ──── 220 Ω ──────── B (синий анод)
GND ────────────────── K (общий катод — самая длинная нога)

Если у вас общий анод (common anode):
  5V ─────────────────── A (общий анод)
  и установите LED_COMMON_ANODE=true в .env
```

> **Почему пины D2, D3, D5?** На Arduino Mega 2560 библиотека `Servo` использует Timer 1, который отключает ШИМ на пинах D11 и D12. Пины D2, D3, D5 управляются Timer 3 — они полностью совместимы с Servo и поддерживают `analogWrite`.

## 13. Сборка дистрибутива .exe

### Предварительные требования

- Poetry с установленными dev-зависимостями (`poetry install`)
- Это нужно делать только один раз или после изменения кода

### Сборка

```bash
poetry run python build.py
```

Скрипт выполняет:
1. Запускает PyInstaller с файлом `grow_control.spec`
2. Копирует `.env.example` в корень `dist/grow_control/` для удобства пользователя

**Результат**: папка `dist/grow_control/` (≈124 МБ) со структурой:

```
dist/grow_control/
├── grow_control.exe          ← запускаемый файл
├── .env.example              ← шаблон конфигурации (переименуйте в .env)
└── _internal/                ← вспомогательные библиотеки Qt, Python, tzdata…
```

### Почему не один файл, а папка?

PySide6 (Qt) включает сотни DLL-файлов, плагинов и переводов. При сборке в один файл (onefile) PyInstaller распаковывает всё это во временную папку при каждом запуске, что даёт задержку 15–30 секунд. Формат **одна папка** (onedir) запускается мгновенно и является стандартом для Qt-приложений.

## 14. Запуск готового .exe (конечный пользователь)

### Первый запуск

1. Скопировать папку `dist/grow_control/` в любое удобное место (например, `C:\Programs\GrowControl\`).

## 19. Часто задаваемые вопросы (FAQ)

### Вопросы по запуску

**Q: Приложение не находит Arduino автоматически.**

A: Проверьте в Диспетчере устройств → Порты (COM и LPT). Если Arduino не виден вообще — установите драйвер CH340 (для большинства клонов) или выполните поиск драйвера. Если виден, но не определяется автоматически — укажите явно: `SERIAL_PORT=COM3` в `.env`.

**Q: Приложение зависает при подключении.**

A: Вероятно, на плате не прошит `grow_control.ino`. Откройте Serial Monitor в Arduino IDE — если вместо `GROW_CTRL_READY` идут бинарные данные, то прошита StandardFirmata или другой скетч. Загрузите `grow_control.ino`.

**Q: DHT11 не возвращает данные (поля t и h отсутствуют).**

A: 1) Проверьте питание: 3.3 В или 5 В на VCC DHT11. 2) Убедитесь что DATA-пин подключён к D4 Arduino. 3) Обязательно установите подтягивающий резистор 10 кΩ между DATA и VCC. 4) Некоторые DHT11-модули имеют встроенный резистор — в этом случае внешний не нужен.

**Q: Значение влажности почвы всегда 100% или всегда 0%.**

A: Датчик нужно откалибровать. Смотрите поле `s` в JSON через Serial Monitor: 1) Опустите датчик в воду — запишите значение (`SOIL_ADC_WET`). 2) Выньте, вытрите, подождите — запишите значение на воздухе (`SOIL_ADC_DRY`). 3) Внесите в `.env`.

**Q: Вентилятор работает, но GUI не показывает "Fan: ON".**

A: В `MOCK_HARDWARE=false` режиме состояние вентилятора управляется через `Controller._check_temperature()`. Убедитесь что температура действительно превысила порог. При `MOCK_HARDWARE=true` вентилятор логируется в терминал, но физически не включается.

### Вопросы по архитектуре (для защиты диплома)

**Q: Почему выбран Python, а не C++ или Java?**

A: Python предоставляет обширную экосистему библиотек (PySide6, pyserial, pydantic) и позволяет сосредоточиться на бизнес-логике. Для данной задачи (опрос датчиков раз в 2 секунды, отображение данных) производительность Python полностью достаточна. Java потребовала бы значительно более сложного GUI-кода; C++ — ручного управления памятью.

**Q: Чем обоснован выбор PySide6, а не Tkinter или wxPython?**

A: PySide6 — официальные Qt-биндинги с нативным внешним видом на Windows, богатым набором виджетов (`QSpinBox`, `QTimeEdit`, `QSplitter`, `QPlainTextEdit`) и встроенной системой сигналов/слотов. Tkinter выглядит устаревшим на Windows 10/11. wxPython имеет хуже документацию и меньшее сообщество.

**Q: Почему протокол JSON, а не бинарный?**

A: JSON читаем человеком (что критически важно при отладке через Serial Monitor), легко парсится в Python одной строкой (`json.loads()`), расширяем (можно добавить новые поля, не ломая совместимость) и стандартизирован. Накладные расходы (несколько байт на строку) несущественны при скорости 115200 бод.

**Q: Как обеспечена потокобезопасность?**

A: Фоновый поток `serial-read` записывает в `_latest` через `threading.Lock`. `Controller.run()` читает `_latest` через `get_latest_reading()` — тот же Lock гарантирует атомарность чтения. GUI обновляется только через Qt-сигналы, которые Qt маршрутизирует в основной поток автоматически.

**Q: Что такое гистерезис и зачем он нужен?**

A: Гистерезис — это разница между порогом включения и порогом выключения устройства. Без него при температуре точно на пороге (например, 28.0 °C) вентилятор мерцал бы: включился бы при 28.01, выключился при 27.99, снова включился... Это называется "дребезгом управления". Гистерезис в 1–2 °C создаёт "мёртвую зону" и устраняет проблему.

**Q: Как реализован Light Scheduler и почему важен часовой пояс?**

A: `LedScheduler` использует библиотеку `zoneinfo` и базу данных IANA для определения текущего локального времени с учётом перехода на летнее время. Расчёт `should_be_on` корректно обрабатывает "ночные окна" (когда время включения > времени выключения, например, 20:15 → 06:30 — переход через полночь). Без явного часового пояса при переводе часов расписание могло бы сбиться.

**Q: Что такое Protocol в Python и зачем он применяется?**

A: `typing.Protocol` — механизм структурной типизации (duck typing с проверкой типов). `BoardGateway` объявлен как Protocol: любой класс с методами `connect()`, `disconnect()`, `get_latest_reading()` и т.д. автоматически считается его реализацией без явного наследования. Это позволяет `Controller` работать одинаково с `SerialBoard` и `MockBoard` без условных операторов, а `mypy` — проверять совместимость.

**Q: Почему настройки хранятся в двух местах — `.env` и `settings.json`?**

A: Это два принципиально разных вида настроек:
- `.env` — **конфигурация инфраструктуры**: какой COM-порт, какие пины, уровень логов. Меняется редко, задаётся при установке.
- `settings.json` — **пользовательские настройки бизнес-логики**: пороги температуры, расписание LED. Меняются часто, сохраняются автоматически через GUI.

**Q: Как работает PyInstaller? Почему дистрибутив — папка, а не один файл?**

A: PyInstaller анализирует граф импортов (`main.py` и все его зависимости), собирает Python-байткод в архив, копирует нужные DLL Qt и Python, и создаёт загрузчик (`bootloader`). При запуске загрузчик инициализирует Python, распаковывает архив и вызывает `main()`. Формат "папка" (onedir) выбран для мгновенного старта: все DLL уже на месте. Формат "один файл" (onefile) требует распаковки во временную папку при каждом запуске — для Qt-приложения это 15–30 секунд.
